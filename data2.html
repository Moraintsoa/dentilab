🔴 Problèmes critiques
Critique #1 — Sécurité RGPD / HDS USERS.role est un simple string libre. Aucune contrainte d'enum. Un bug ou une injection peut élever les privilèges. Il faut un enum strict (docteur, assistante, secretaire, admin) avec contrôle applicatif + RLS PostgreSQL par cabinet_id.
Critique #2 — Données sensibles non protégées PATIENTS stocke numéro de sécu, mutuelle, antécédents médicaux en clair. Aucune mention de chiffrement colonne (pgcrypto). En environnement HDS (Hébergeur Données de Santé), c'est une obligation légale française.
Critique #3 — Aucun audit log global DENTAL_CHART_HISTORY existe seulement pour le schéma dentaire. Toutes les autres tables (consultations, traitements, factures) n'ont aucune traçabilité des modifications. En médecine, toute modification doit être tracée (qui, quand, avant/après).
🟡 Avertissements logiques
Avert. #1 — DENTAL_CHARTS mal modélisée Une ligne par dent par patient = potentiellement 32 lignes par patient, difficile à requêter atomiquement. La colonne tooth_number devrait être contrainte (1–48 FDI). De plus, il manque le numéro de quadrant FDI et les faces atteintes (mésiale, distale, occlusale…) qui sont centrales en clinique dentaire.
Avert. #2 — TREATMENTS flottants Un traitement peut exister sans lien avec un TREATMENT_PLAN. Il n'y a pas de FK entre TREATMENTS et TREATMENT_PLAN_ITEMS. Si un acte planifié est réalisé, il n'y a aucun moyen de le marquer comme tel automatiquement.
Avert. #3 — APPOINTMENTS sans confirmation patient Pas de colonne pour le canal de rappel (sms, email), pas de confirmation de présence patient, ni de motif d'annulation. Essentiels pour la gestion d'agenda réelle.
Avert. #4 — Pas de versioning de prix TREATMENTS.unit_price est modifiable. Si le prix change, les anciennes factures sont corrompues. Il faut figer le prix au moment de la facturation (ce que INVOICE_ITEMS.unit_price fait partiellement, mais TREATMENT_PLAN_ITEMS.estimated_price n'est pas relié).
🟢 Tables manquantes
Manque #1 — NOTIFICATIONS Aucune table pour gérer les rappels de RDV (SMS/email), les alertes de stock bas, les relances de factures impayées. Sans ça, tout passe en dur dans le code ou dans un service externe non tracé.
Manque #2 — PRESCRIPTIONS Les prescriptions médicamenteuses (antibiotiques, antidouleurs) ne sont pas modélisées. Un champ text dans CONSULTATIONS n'est pas suffisant — il faut un historique, un lien avec le patient et une date de validité.
Manque #3 — AUDIT_LOGS global Une table centrale AUDIT_LOGS (table_name, record_id, action, old_json, new_json, user_id, ip, at) est indispensable pour la conformité médicale. Elle peut être alimentée par des triggers PostgreSQL automatiques.
Manque #4 — CONSENTEMENTS RGPD Aucune trace du consentement éclairé patient (date, version du formulaire, canal de signature). Obligatoire RGPD + réglementation médicale française.
Manque #5 — REPORTS / RAPPORTS Pas de table pour stocker les rapports générés (CA mensuel, actes fréquents…). Sans ça, les rapports sont recalculés à chaque requête, ce qui est lent sur des données volumineuses.
✅ Ce qui est bien fait
✓ Séparation INVOICE / INVOICE_ITEMS Bonne normalisation. Le pattern entête-lignes est correct et permet des factures multi-actes proprement.
✓ INVENTORY_MOVEMENTS + INVENTORY_USAGE Tracer les mouvements de stock ET lier leur consommation aux traitements est une excellente décision de modélisation. Rare de le voir dans un premier jet.
✓ TREATMENT_PLANS séparé des TREATMENTS La distinction plan planifié vs acte réalisé est conceptuellement juste. Beaucoup d'architectures fusionnent les deux à tort.
✓ DENTAL_CHART_HISTORY Versionner l'état des dents est une excellente initiative clinique. À étendre à l'ensemble du système avec un audit_log global.
✓ Multi-tenant via cabinet_id La présence de cabinet_id sur les tables clés est la bonne approche pour un SaaS. Il faut maintenant y ajouter le RLS PostgreSQL.

1
Row Level Security (RLS) PostgreSQL — priorité absolue
Active l'isolation des données par cabinet directement au niveau de la BDD. Chaque requête SQL est automatiquement filtrée, même si le code applicatif oublie un WHERE cabinet_id = ?. Sans ça, une faille applicative peut exposer TOUS les patients de TOUS les cabinets.
-- Activer RLS sur chaque table multi-tenant ALTER TABLE patients ENABLE ROW LEVEL SECURITY; CREATE POLICY "isolation_cabinet" ON patients USING (cabinet_id = current_setting('app.current_cabinet_id')::uuid); -- Idem pour: appointments, consultations, treatments, -- invoices, dental_charts, radiographies, inventory_items...
2
Enum strict sur USERS.role — sécurité d'accès
Remplacer string libre par un type PostgreSQL enum. Cela empêche tout rôle arbitraire d'être inséré, et permet de construire des politiques RLS basées sur le rôle sans ambiguïté.
CREATE TYPE user_role AS ENUM ( 'docteur', 'assistante', 'secretaire', 'admin' ); ALTER TABLE users ALTER COLUMN role TYPE user_role USING role::user_role;
3
Chiffrement colonnes sensibles — conformité HDS
Les données de santé (N° sécurité sociale, mutuelle, antécédents) doivent être chiffrées au repos. PostgreSQL pgcrypto permet de chiffrer directement en BDD avec une clé applicative, même si le disque est compromis.
-- Extension pgcrypto CREATE EXTENSION pgcrypto; -- Chiffrer au INSERT INSERT INTO patients (numero_securite, ...) VALUES (pgp_sym_encrypt('1234567890123', 'APP_SECRET_KEY'), ...); -- Déchiffrer à la lecture SELECT pgp_sym_decrypt(numero_securite::bytea, 'APP_SECRET_KEY') FROM patients WHERE id = ?;
4
AUDIT_LOGS via trigger automatique
Créer un trigger générique qui s'applique à toutes les tables sensibles. Chaque INSERT/UPDATE/DELETE est automatiquement loggé avec l'état avant et après, sans que le code applicatif ait besoin de s'en occuper.
CREATE OR REPLACE FUNCTION fn_audit_log() RETURNS TRIGGER AS $$ BEGIN INSERT INTO audit_logs (table_name, record_id, action, old_data, new_data, user_id, created_at) VALUES ( TG_TABLE_NAME, COALESCE(NEW.id, OLD.id), TG_OP, to_jsonb(OLD), to_jsonb(NEW), current_setting('app.current_user_id')::uuid, NOW() ); RETURN COALESCE(NEW, OLD); END; $$ LANGUAGE plpgsql; -- Appliquer sur les tables critiques CREATE TRIGGER trg_audit_consultations AFTER INSERT OR UPDATE OR DELETE ON consultations FOR EACH ROW EXECUTE FUNCTION fn_audit_log();
5
Lien TREATMENTS → TREATMENT_PLAN_ITEMS
Ajouter une FK nullable treatment_plan_item_id dans TREATMENTS. Quand un acte planifié est réalisé, la FK est remplie et le statut du TREATMENT_PLAN_ITEM passe automatiquement à realise via trigger. Ça ferme la boucle plan↔réalisation.
-- FK nullable (un traitement peut être hors plan) ALTER TABLE treatments ADD COLUMN treatment_plan_item_id uuid REFERENCES treatment_plan_items(id); -- Trigger: marquer le plan item comme réalisé CREATE OR REPLACE FUNCTION fn_mark_plan_item_done() RETURNS TRIGGER AS $$ BEGIN IF NEW.treatment_plan_item_id IS NOT NULL THEN UPDATE treatment_plan_items SET status = 'realise' WHERE id = NEW.treatment_plan_item_id; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql;
6
DENTAL_CHARTS — contrainte FDI + faces atteintes
Ajouter une contrainte CHECK sur fdi_number pour valider la notation FDI (11–18, 21–28, 31–38, 41–48). Ajouter faces_atteintes text[] pour stocker la précision clinique (mésiale, distale, occlusale, buccale, linguale) essentielle en dentisterie.
CREATE TYPE tooth_face AS ENUM ( 'mesiale', 'distale', 'occlusale', 'buccale', 'linguale', 'palatine', 'incisale' ); CREATE TYPE tooth_status_enum AS ENUM ( 'saine', 'caries', 'obturee', 'couronne', 'absente', 'bridge', 'implant', 'devitalisee', 'surveillance', 'fracture' ); ALTER TABLE dental_charts ADD COLUMN fdi_number int CHECK ( (fdi_number BETWEEN 11 AND 18) OR (fdi_number BETWEEN 21 AND 28) OR (fdi_number BETWEEN 31 AND 38) OR (fdi_number BETWEEN 41 AND 48) ), ADD COLUMN faces_atteintes tooth_face[], ADD COLUMN tooth_status tooth_status_enum NOT NULL DEFAULT 'saine';
7
Snapshot prix dans TREATMENTS — intégrité financière
Ajouter unit_price_snapshot figé au moment où l'acte est enregistré. Si le tarif évolue ensuite, les consultations passées restent cohérentes. C'est le même principe qu'INVOICE_ITEMS.unit_price, à appliquer dès le traitement.
ALTER TABLE treatments ADD COLUMN unit_price_snapshot decimal(10,2) NOT NULL DEFAULT 0, ADD COLUMN currency char(3) NOT NULL DEFAULT 'EUR'; -- Au moment de l'INSERT d'un treatment, snapshotter le prix courant -- depuis un référentiel d'actes (table CARE_CATALOG à ajouter) INSERT INTO treatments (consultation_id, treatment_type, unit_price_snapshot, ...) SELECT ..., catalog.current_price, ... FROM care_catalog catalog WHERE catalog.code = ?;
